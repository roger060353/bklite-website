# 快速入门

## 一、前置条件

在开始使用告警中心前，建议先确认以下准备项已经完成：

*   已具备可访问 BK-Lite 平台的账号，并拥有告警中心相关模块的查看与配置权限。
*   已在「集成」中创建或确认可用的告警源，并获取对应的接入密钥。
*   已明确需要接入的事件字段，至少建议包含标题、等级、发生时间、资源信息或业务标识等关键上下文。
*   已明确本次希望验证的场景，例如基础设施异常聚合、流水线失败跟踪或业务任务异常通知。

> **注意：**
> 告警中心的效果高度依赖事件字段质量。若上送事件缺少资源标识、来源标识或业务上下文，后续的聚合、分派与恢复判断会受到影响。

## 二、第一步：定义事件聚合规则

路径：配置 -> 相关性规则

相关性规则决定 Event 如何被聚合为 Alert，是整套告警治理流程的起点。建议先为一个明确场景配置规则，例如主机性能异常、数据库异常或流水线失败事件。

> **界面指引：**
>
> ![相关性规则](https://static.cwoa.net/a9437b34f4a845afac9ade192f5e64b7.png)
>
> *   **图表解读/配置逻辑**：在这个页面中，用户需要重点关注规则匹配范围、分组字段、窗口类型和关闭时间。它们共同决定了哪些事件会进入同一条告警，以及告警在何时恢复或自动结束。

配置时建议重点关注以下内容：

*   **匹配规则**：定义哪些事件进入当前规则处理范围。
*   **分组字段**：定义事件聚合时按哪些维度归并，例如资源实例、服务名或位置。
*   **窗口类型**：定义规则在时间维度上的生效方式。
*   **关闭时间**：定义该规则生成的告警在长期未处理时的自动关闭时长。

窗口类型说明：

*   **固定窗口**：适合对固定时间片内的数据进行统计或聚合。
*   **滑动窗口**：适合连续时间段内的异常识别，窗口之间允许重叠。
*   **会话窗口**：适合基于事件间隔识别问题持续性的场景，例如失败后长时间未恢复。

## 三、第二步：推送事件到告警中心

路径：集成 -> 告警源 -> 指南

在创建并启用告警源后，可以按照接入指南向告警中心推送事件。当前平台支持 REST API 与 NATS 等方式，下面以 REST API 为例说明最小接入方式。

```bash
curl -X POST 'https://xxxxxxxxx/api/proxy/alerts/api/receiver_data' \
  -H 'Content-Type: application/json' \
  -H 'SECRET: 从集成中的对应告警源获取' \
  -d '{
    "source_id": "restful",
    "events": [
      {
        "item": "mem_usage",
        "level": "1",
        "title": "主机 10.10.24.62 的内存使用率过高",
        "value": 95,
        "labels": {
          "resource_name": "10.10.24.62",
          "resource_type": "host"
        },
        "description": "主机 10.10.24.62 内存使用率持续超过阈值"
      }
    ]
  }'
```

建议上送时至少包含以下信息：

*   **事件标题**：便于值班人员快速理解问题。
*   **事件等级**：用于后续筛选、分派和提醒频率判断。
*   **资源信息**：例如 `resource_name`、`resource_type`，用于聚合和定位。
*   **描述信息**：补充上下文，帮助后续处置与复盘。

> **界面指引：**
>
> ![告警源](https://static.cwoa.net/d77279aa03604616947a03c7a395f191.png)
>
> *   **图表解读/配置逻辑**：集成中心不仅用于管理接入方式，也用于核验事件是否已进入平台。用户可在告警源详情中查看接入指南、鉴权参数和最近事件情况，确认接入链路是否正常。

## 四、第三步：验证事件是否成功入站

路径：事件

完成推送后，建议先进入事件页面核验原始事件是否已成功进入平台。这里可以查看来源、等级、状态和事件内容，快速判断问题出在“接入”还是“聚合规则”。

验证重点包括：

*   事件是否来自预期的告警源。
*   事件等级、标题和资源信息是否解析正确。
*   事件是否因屏蔽策略而未进入后续链路。
*   时间戳是否合理，避免因为时间异常导致规则判断偏差。

## 五、第四步：查看生成的告警并开始处理

路径：告警

当事件命中规则并满足窗口条件后，平台会生成对应告警。你可以在告警页面查看生成结果，并进入详情页查看关联事件、处理记录与通知状态。

> **界面指引：**
>
> ![告警列表](https://static.cwoa.net/2fcc8ed971d8426d85f4e1eb35493cab.png)
>
> *   **图表解读/配置逻辑**：告警页面是日常处理的主入口。用户既可以从列表中快速筛选高优先级问题，也可以进入详情核查这条告警由哪些事件聚合而来，再决定是认领、转派、关闭还是升级为事故。

在这个阶段，建议重点验证以下内容：

*   告警是否按照预期聚合，没有出现明显重复。
*   告警级别、来源和标题是否符合预期。
*   详情中是否能看到关联事件，便于回溯聚合效果。
*   分派、认领与关闭等操作是否符合团队预期流程。

## 六、第五步：将重大问题升级为事故

路径：告警 -> 选择告警 -> 升级为事故

当某条告警已具备明显业务影响，或者需要多个角色共同处理时，可以将其升级为 Incident。升级后，团队可以在事故页面统一跟踪问题进展和关联告警。

> **界面指引：**
>
> ![升级为事故](https://static.cwoa.net/794163529d894627b151deebdc6f1f7f.png)
>
> *   **图表解读/配置逻辑**：升级为事故的目的不是简单改变名称，而是将单点告警转化为更高等级的协同对象。这里需要补充事故标题、等级和参与人员，使后续处理具备更明确的组织方式。

事故创建完成后，可前往事故页面查看和继续处理：

> **界面指引：**
>
> ![事故列表](https://static.cwoa.net/a00cb4152b81489f873f6336679706a9.png)
>
> *   **图表解读/配置逻辑**：事故页面用于统一跟踪已升级的问题。用户可以查看事故状态、关联告警和处理进展，适合处理中高影响等级的问题。

## 七、结果验证与闭环建议

完成以上步骤后，建议按以下方式验证本次接入是否形成完整闭环：

*   在事件页面确认原始事件已成功入站。
*   在告警页面确认规则已正确聚合并形成可处理对象。
*   在详情中确认可回溯到关联事件和处理记录。
*   在需要的场景下验证自动分派、通知或升级事故流程是否按预期生效。

如果后续需要提升告警治理质量，建议继续完善以下配置：

*   为高频问题补充分派策略，减少人工转派。
*   为维护窗口或已知低价值事件补充屏蔽策略，进一步降低噪声。
*   根据实际处置时长优化提醒频率和自动关闭策略。
