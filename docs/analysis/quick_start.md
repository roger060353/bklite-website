# 快速入门

## 一、前置条件

在开始使用 BK-Lite 运营分析前，建议先确认以下准备项已经完成：

*   已具备可访问 BK-Lite 平台的账号，并拥有运营分析模块对应的查看与配置权限。
*   平台内置的数据源与命名空间已由管理员完成初始化，能够满足当前常规使用场景。
*   已明确本次希望验证的分析对象，例如资源总览、告警趋势、服务拓扑或架构盘点。
*   已确认需要对哪些团队开放该分析资产，便于后续在目录与页面层面设置归属范围。

> **注意：**
> BK-Lite 运营分析默认已提供可直接使用的内置数据源与命名空间。正常情况下，普通用户无需进入设置页调整底层接入；只有在现有内置能力无法满足业务需求时，才建议由管理员补充或修改相关配置。

## 二、第一步：规划目录并创建分析资产

路径：视图 -> 左侧目录树 -> 「+」或操作菜单

进入运营分析后，建议先在视图页规划目录结构，再创建具体的仪表盘、拓扑图或架构图对象。目录树支持多层级组织方式，适合按业务系统、专题场景或治理目标沉淀分析内容。

建议优先建立以下其中一种组织方式：

*   按业务域组织，例如电商、支付、履约、基础平台。
*   按分析专题组织，例如稳定性总览、容量治理、告警复盘、架构盘点。
*   按团队职责组织，例如平台运维、业务运维、SRE。

> **界面指引：**
>
> 当前建议补充「视图页目录树」截图，用于展示左侧目录树、新建入口和内容区切换关系。
>
> *   **图表解读/配置逻辑**：目录树不是简单导航栏，而是团队分析资产的组织入口。先把目录结构规划清楚，后续新增的仪表盘、拓扑图和架构图才更容易复用与沉淀。

在创建目录和分析对象时，建议注意：

*   目录适合放长期沉淀的分析主题，不建议把一次性任务也固化为独立层级。
*   仪表盘适合表达趋势、占比和关键数值。
*   拓扑图适合表达对象关系、依赖结构和节点状态。
*   架构图适合表达资源结构、系统分层和平台拓扑。

## 三、第二步：创建首个仪表盘或画布

路径：视图 -> 选择目录 -> 新建仪表盘 / 新建拓扑图 / 新建架构图

如果是第一次验证，建议优先从仪表盘开始，因为它最适合快速确认内置数据源是否满足当前分析需求；如果你的目标是表达依赖关系或状态分布，则可以直接从拓扑图开始。

配置时建议重点关注以下内容：

*   **优先复用内置数据源**：直接从平台已提供的数据源中选择合适能力，无需从零配置底层接入。
*   **按页面目标选择载体**：趋势总览优先用仪表盘，关系定位优先用拓扑图，结构表达优先用架构图。
*   **合理补充参数**：按现有数据源模板补充必要过滤项，尤其是时间范围、对象范围和分类条件。
*   **选择合适图表类型**：趋势类用折线图，分类对比用柱状图，占比类用饼图，关键状态用单值图。
*   **控制首页复杂度**：首个页面建议先做最小可用版本，不要一次叠加过多组件或节点。

> **界面指引：**
>
> 当前建议补充「新建仪表盘 / 拓扑图 / 架构图」截图，用于展示对象创建入口和首个页面的配置过程。
>
> *   **图表解读/配置逻辑**：快速入门阶段的重点不是搭底座，而是先用现成能力做出第一个可用页面，验证平台是否能支撑当前团队的分析表达需求。

### 仪表盘配置建议

*   先添加一个组件并选择数据源。
*   按内置数据源模板补充必要参数，尤其是时间范围与过滤项。
*   选择最符合当前场景的图表类型，例如趋势类用折线图，分类对比用柱状图，占比类用饼图，关键状态用单值图。
*   若页面内存在多个时间型组件，建议统一启用全局时间选择范围，便于做横向对比。

### 拓扑图配置建议

*   先创建最小可用画布，只放核心节点，不要一开始铺满全部对象。
*   对需要动态展示状态的节点，可绑定内置单值数据源并配置阈值展示。
*   对需要解释关系的对象，可补充连线、标签和文本节点，提升可读性。
*   若需要持续观察变化，可配置刷新频率，让关键状态定时更新。

### 架构图配置建议

*   先确定画布要表达的是“资源盘点”还是“逻辑架构”，避免同一张图承担过多目标。
*   优先使用统一图标体系，保证团队内部表达口径一致。
*   对跨云、跨平台场景，可分层展示不同类型资源，避免一屏过载。

> **界面指引：**
>
> 当前建议补充「视图页」截图，用于展示左侧目录树、右侧内容区以及仪表盘/拓扑图/架构图的切换关系。
>
> *   **图表解读/配置逻辑**：视图页是运营分析的主工作台。左侧负责组织资产，右侧负责编辑和查看内容，二者配合后才形成真正可复用的分析空间。

## 四、第三步：保存并回看分析结果

路径：视图 -> 当前分析对象 -> 保存

完成组件或画布配置后，应及时保存并回看结果，确认页面是否已经具备团队共享价值。这里的验证重点不只是“能不能显示数据”，而是“是否已经具备后续复用与持续维护的基础”。

建议重点检查以下内容：

*   数据是否来自预期的内置数据源，返回结果是否符合当前分析目标。
*   图表类型是否适合当前分析目标，而不是仅仅能显示。
*   参数是否足够通用，其他人接手时不需要重新理解底层逻辑。
*   目录命名、页面命名和描述是否足够清晰，便于后续检索和复用。
*   团队权限是否设置合理，确保应看的人能看到，不应看的人不会误用。

## 五、结果验证与闭环建议

完成以上步骤后，建议按以下方式确认这次接入已经形成完整闭环：

*   在视图页确认目录结构、分析对象和页面内容已经成功保存。
*   在页面中确认内置数据源返回结果稳定，筛选条件和时间范围能正常工作。
*   切换到其他团队成员视角或权限范围，确认页面共享边界符合预期。
*   以真实场景继续迭代首个看板或画布，而不是停留在一次性验证页面。

如果后续希望进一步提升运营分析的使用效果，建议继续做以下优化：

*   为高频分析场景建立统一的目录命名规范，避免团队资产持续膨胀后难以管理。
*   对重点页面设置固定的维护责任人，确保分析资产长期可用。

## 六、管理员按需扩展说明

在大多数场景下，普通用户直接使用平台内置的数据源与命名空间即可。只有在以下情况下，才建议由管理员进入设置页补充底层配置：

*   现有内置数据源无法覆盖新的业务分析场景。
*   需要新增特定命名空间或对接新的数据环境。
*   需要调整参数模板、图表类型限制或团队可见范围。

此时可按需进入以下页面维护：

*   设置 -> 数据源：维护数据源路径、标签、图表类型和参数模板。
*   设置 -> 命名空间：维护连接信息、TLS 配置和访问环境。
