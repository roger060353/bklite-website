# 快速入门

## 一、前置条件

在开始使用 BlueKing Lite 运营分析前，建议确认以下准备项：

*   **平台账号与权限**：已具备可访问 BlueKing Lite 平台的账号，并拥有运营分析模块的查看权限。如需创建分析资产，需额外申请目录、仪表盘或数据源等操作权限。
*   **内置数据环境**：平台默认已初始化内置命名空间和数据源标签（CMDB、告警、监控、日志、业务、其他），普通用户开箱即用，无需额外配置。
*   **分析目标明确**：建议提前明确本次希望验证的分析场景，例如：
    *   资源总览类（用仪表盘展示主机、应用数量）
    *   趋势分析类（用折线图展示告警趋势）
    *   关系定位类（用拓扑图展示服务依赖）
    *   架构表达类（用架构图展示系统结构）
*   **团队归属确认**：确认需要对哪些团队开放该分析资产，便于后续设置归属范围。

> **注意：**
> 运营分析默认提供可直接使用的内置数据源与命名空间。**普通用户无需进入设置页调整底层接入**；仅当内置能力无法满足业务需求时，才建议由管理员补充或修改相关配置。

## 二、第一步：规划目录并创建分析资产

**操作路径**：视图 → 左侧目录树 → 「+」按钮或右键菜单

进入运营分析后，建议先在视图页规划目录结构，再创建具体的分析对象。合理的目录规划是后续资产沉淀和团队协作的基础。

### 2.1 推荐目录组织方式

| 组织维度 | 示例 | 适用场景 |
|---------|------|---------|
| **按业务域** | 电商 / 支付 / 履约 / 基础平台 | 多业务线独立运维 |
| **按分析专题** | 稳定性总览 / 容量治理 / 告警复盘 / 架构盘点 | 专题化运维分析 |
| **按团队职责** | 平台运维 / 业务运维 / SRE / 安全 | 职责边界清晰 |

### 2.2 创建目录的操作步骤

1. 在视图页左侧目录树空白处或目标目录上右键，选择「新建目录」
2. 填写目录名称（同父目录下名称不可重复）
3. 填写描述信息（可选，建议填写便于后续理解目录用途）
4. 点击保存，目录即创建成功

### 2.3 目录规划注意事项

*   **层级控制**：目录层级上限为 3 层，规划时注意控制深度，避免过度嵌套
*   **命名规范**：建议使用「业务域_场景」或「团队_专题」格式，如「电商_稳定性总览」
*   **目录 vs 分析对象**：目录适合放长期沉淀的分析主题，不要把一次性任务固化为独立层级
*   **删除保护**：目录下存在子目录或分析对象时不可直接删除，需先清空内容

> **界面指引：**
>
> 当前建议补充「视图页目录树」截图，展示左侧目录树、新建入口、右键菜单。
>
> *   **图表解读**：目录树是团队分析资产的组织入口。先把目录结构规划清楚，后续新增的仪表盘、拓扑图和架构图才更容易复用与沉淀。

## 三、第二步：创建首个分析对象

**操作路径**：视图 → 选择目标目录 → 点击「新建仪表盘」/「新建拓扑图」/「新建架构图」

如果是第一次验证，建议**优先从仪表盘开始**，因为它最适合快速确认内置数据源是否满足当前分析需求。如果你的目标是表达依赖关系或系统架构，可直接从拓扑图或架构图开始。

### 3.1 创建仪表盘（推荐首选）

#### 步骤说明

1. **进入创建页面**：在目标目录上点击「新建仪表盘」，填写名称和描述后进入编辑页
2. **添加第一个组件**：
    *   点击画布区域的「添加组件」
    *   在组件配置面板中选择数据源（建议优先选择内置数据源）
    *   系统会自动加载该数据源配置的参数模板
3. **配置参数**：
    *   根据数据源模板补充必要过滤项（如时间范围、对象范围）
    *   固定值参数会自动填充，可编辑参数需手动选择或输入
4. **选择图表类型**：根据数据特性选择合适图表：
    *   **折线图**：趋势类数据（如告警数量随时间变化）
    *   **柱状图**：分类对比类数据（如各业务线主机数量）
    *   **饼图**：占比类数据（如告警级别分布）
    *   **单值图**：关键指标（如总主机数、未恢复告警数）
5. **调整布局**：拖拽组件调整位置和大小，组织成易读的页面结构
6. **保存页面**：点击顶部「保存」按钮，完成首个仪表盘创建

#### 配置建议

*   **优先使用内置数据源**：平台已预置 CMDB、监控、告警等常用数据源，无需从零配置
*   **控制页面复杂度**：首个页面建议先做「最小可用版本」，验证数据正确性后再逐步丰富
*   **启用全局时间**：若页面包含多个时间型组件，建议启用全局时间选择器，便于统一筛选

> **界面指引：**
>
> 当前建议补充「新建仪表盘」截图，展示组件配置面板、数据源选择、参数填写、图表类型选择。
>
> *   **图表解读**：快速入门阶段的重点不是搭底座，而是先用现成能力做出第一个可用页面，验证平台是否能支撑当前团队的分析表达需求。

### 3.2 创建拓扑图

#### 适用场景

*   展示服务之间的调用关系
*   表达网络拓扑结构
*   定位故障影响范围
*   展示告警在系统中的传播路径

#### 步骤说明

1. **进入创建页面**：在目标目录上点击「新建拓扑图」，填写名称和描述后进入画布编辑页
2. **添加核心节点**：
    *   从左侧工具栏拖拽「图标节点」到画布，表示关键服务或资源
    *   使用「文本节点」添加说明文字
3. **建立连接关系**：
    *   选中节点，点击连接点拖拽到目标节点，创建连线
    *   双击连线可添加标签（如调用关系、延迟数值）
4. **绑定状态数据**（可选）：
    *   添加「单值节点」或「图表节点」
    *   在属性面板中绑定数据源，配置阈值配色
    *   设置刷新频率，让节点状态定时更新
5. **优化布局**：
    *   使用「适配画布」按钮调整整体可视区域
    *   使用「撤销/重做」调整不满意的操作
6. **保存页面**：点击顶部「保存」按钮

#### 配置建议

*   **先画最小可用图**：首版只放核心节点（如网关、核心服务、数据库），验证思路后再扩展
*   **善用标签**：连线上添加标签说明关系类型，提升图谱可读性
*   **动态节点聚焦**：对需要重点监控的节点绑定单值数据源，用颜色区分健康状态

---

### 3.3 创建架构图

#### 适用场景

*   系统架构评审与展示
*   资源盘点与上云规划
*   技术债务梳理
*   跨团队技术方案沟通

#### 步骤说明

1. **进入创建页面**：在目标目录上点击「新建架构图」，填写名称和描述后进入画布编辑页
2. **选择图标体系**：
    *   从左侧图标库选择合适图标（云资源、平台组件、CMDB 资源等）
    *   拖拽到画布并调整位置
3. **组织层次结构**：
    *   使用「基础图形」绘制分层边界（如网络层、应用层、数据层）
    *   通过颜色区分环境（如生产环境红色、测试环境绿色）
4. **添加说明元素**：
    *   使用「文本节点」添加分层说明、注意事项
    *   使用连线和标签表达组件间关系
5. **保存页面**：点击顶部「保存」按钮

#### 配置建议

*   **先确定表达目标**：明确是展示「逻辑架构」还是「物理部署」，避免同一张图承担过多目标
*   **统一图标口径**：同一类资源使用相同图标，保证团队内部表达一致
*   **分层展示**：跨云、跨平台场景建议分层展示，避免单屏信息过载

> **界面指引：**
>
> 当前建议补充「视图页」截图，展示左侧目录树、右侧内容区以及仪表盘/拓扑图/架构图的切换关系。
>
> *   **图表解读**：视图页是运营分析的主工作台。左侧负责组织资产，右侧负责编辑和查看内容，二者配合后才形成真正可复用的分析空间。

## 四、第三步：保存并验证分析结果

**操作路径**：视图 → 当前分析对象 → 保存 → 切换查看模式验证

完成组件或画布配置后，应及时保存并验证结果，确认页面是否已经具备团队共享价值。

### 4.1 保存前检查清单

| 检查项 | 检查内容 | 通过标准 |
|-------|---------|---------|
| **数据来源** | 组件绑定的数据源是否正确 | 数据来自预期的内置或自定义数据源 |
| **数据准确性** | 返回结果是否符合分析目标 | 数值、趋势与预期一致 |
| **图表适配性** | 图表类型是否适合当前数据 | 趋势用折线、对比用柱状、占比用饼图 |
| **参数通用性** | 参数配置是否足够通用 | 其他用户接手时无需重新理解底层逻辑 |
| **命名清晰度** | 目录、页面、组件命名是否清晰 | 通过名称即可理解用途，无需额外说明 |
| **权限合理性** | 团队归属和权限是否设置合理 | 应看的人能看到，不应看的人不可见 |

### 4.2 验证与迭代

1. **数据验证**：切换不同时间范围、过滤条件，确认数据响应正常
2. **权限验证**：切换到其他团队成员账号（或咨询管理员），确认可见范围符合预期
3. **场景验证**：邀请同事一起查看，收集反馈并迭代优化
4. **持续维护**：为重点页面设置维护责任人，确保分析资产长期可用

## 五、结果验证与闭环建议

完成以上步骤后，建议按以下方式确认已经形成完整闭环：

*   **资产沉淀验证**：在视图页确认目录结构、分析对象已成功保存，可在目录树中正常浏览
*   **数据查询验证**：在页面中确认数据源返回结果稳定，筛选条件和时间范围能正常工作
*   **权限边界验证**：切换到其他团队成员视角，确认页面共享边界符合预期
*   **场景落地验证**：以真实运维场景（如值班巡检、故障复盘）试用首个看板或画布，而非停留在一次性验证页面

### 进阶优化建议

如果首个分析对象已验证可用，建议继续以下优化：

*   **建立命名规范**：为高频分析场景建立统一的目录命名规范，如「业务域_场景_视角」
*   **设置维护责任人**：为重点页面指定维护责任人，定期更新数据范围与展示逻辑
*   **沉淀模板**：将通用分析模式保存为团队模板，便于快速复制创建类似页面
*   **权限分级**：根据团队分工细化权限配置，如业务团队仅能查看业务域分析资产

## 六、管理员扩展指南

在大多数场景下，普通用户直接使用平台内置的数据源与命名空间即可。**仅在以下情况下，建议由管理员进入设置页补充底层配置**：

*   现有内置数据源无法覆盖新的业务分析场景（如需要接入第三方业务系统数据）
*   需要新增特定命名空间对接新的数据环境（如新增集群、新增区域）
*   需要调整参数模板、图表类型限制或团队可见范围

### 6.1 创建命名空间

**操作路径**：设置 → 命名空间 → 新建

| 配置项 | 说明 | 示例 |
|-------|------|------|
| 名称 | 命名空间的显示名称 | 生产环境集群A |
| NATS命名空间 | 消息主题前缀 | bklite_prod |
| 账号 | NATS连接账号 | ops_analysis |
| 密码 | NATS连接密码（自动加密） | ***** |
| 域名 | NATS服务器地址 | nats.example.com:4222 |
| 启用TLS | 是否使用TLS加密连接 | 是（生产环境建议开启） |

> **安全提示：**
> 生产环境强烈建议启用 TLS，密码在保存时会自动加密，数据库中不存储明文。

### 6.2 创建数据源

**操作路径**：设置 → 数据源 → 新建

| 配置项 | 说明 | 示例 |
|-------|------|------|
| 名称 | 数据源的显示名称 | 主机资源统计 |
| REST API | 数据接口路径 | /api/cmdb/hosts/statistics |
| 命名空间 | 关联的数据来源 | 生产环境集群A |
| 标签 | 数据源分类标签 | CMDB |
| 图表类型 | 支持的展示形式 | 折线图、柱状图、单值图 |
| 参数模板 | 接口所需参数 | time_range、app_id |

> **配置建议：**
> 参数模板建议按「固定值」与「可编辑值」区分，固定值在配置时确定，可编辑值留给使用者在分析页面调整。

## 七、常见问题（FAQ）

**Q1：为什么我的仪表盘没有数据？**

*   检查数据源是否已启用
*   检查命名空间连接是否正常（可咨询管理员）
*   检查参数配置是否正确，特别是时间范围
*   检查当前用户是否有该数据源所属团队的查看权限

**Q2：如何分享我创建的仪表盘给团队其他人？**

*   确保该仪表盘的「所属团队」设置正确
*   确认团队成员具备「仪表盘查看」权限
*   团队成员可在视图页的目录树中看到该仪表盘

**Q3：拓扑图和架构图有什么区别？**

*   **拓扑图**：侧重表达动态关系和实时状态，支持数据绑定和定时刷新，适合故障定位和运行态观察
*   **架构图**：侧重表达静态资源结构和系统分层，适合架构评审和方案沟通

**Q4：目录层级上限可以调整吗？**

*   当前目录层级上限固定为 3 层，这是为了平衡灵活性与可维护性。如需更深层级，建议通过命名规范来体现层级关系。

**Q5：数据源关联多个命名空间时，数据如何展示？**

*   系统会并行向各命名空间取数，并按命名空间名称分区返回。前端组件可选择展示全部数据或仅展示特定命名空间数据。
